WHAT WAS WRONG ABOUT THE WEBKIT SUMMARY
---------------------------------------

nixpkgs's `pkgs.playwright-driver` ships **WPE WebKit** (the embedded variant,
designed for set-top boxes / kiosks). Launched directly from the normal
`mkShell`, WPE aborts at startup with:

    Could not create WPE EGL display: EGL_SUCCESS. Aborting...

That part was true, but the conclusion was too strong. It is not simply a
container/hardware limitation: the same Nix-packaged WebKit passes in the same
container when launched from a Nix `buildFHSEnv` wrapper with the missing
graphics/media library layout available.

Things that did NOT fix direct `mkShell` execution:
  - WEBKIT_DISABLE_DMABUF_RENDERER=1
  - EGL_PLATFORM=surfaceless + WPE_BACKEND=fdo
  - LIBGL_ALWAYS_SOFTWARE=1
  - Adding pkgs.mesa for software EGL (llvmpipe)

Things that DID fix it:
  - Launch the Node browser tests through a Nix-built FHS runtime wrapper.
  - Include the WebKit graphics/media dependency closure in that FHS runtime.

WebKitGTK + Xvfb is therefore not proven necessary for this harness.


CURRENT NIX-NATIVE PLAYWRIGHT DESIGN
------------------------------------

All in `javascript/flake.nix`:

  packages = [
    pkgs.nodejs_22 pkgs.bun pkgs.deno pkgs.go-task pkgs.just
    pkgs.playwright          # the npm module (playwright-core 1.56.1)
    pkgs.playwright-driver   # browsers + system libs
  ];

  shellHook exports:
    - PLAYWRIGHT_NPM_PATH         (Nix store path of the npm module)
    - PLAYWRIGHT_BROWSERS_PATH    (Nix store path of browser bundle)
    - PLAYWRIGHT_SKIP_VALIDATE_HOST_REQUIREMENTS=true
    - PLAYWRIGHT_FHS              (Linux-only FHS runtime wrapper)
    - npm_config_prefer_offline=true

`javascript/library/package.json` does NOT declare `playwright`. There is
ONE source of truth for the version: nixpkgs.

`javascript/library/Taskfile.yml` setup symlinks the Nix-store playwright
into `node_modules/playwright` so node ESM resolution finds it
(NODE_PATH is ignored for ESM):

    npm ci
    mkdir -p node_modules
    ln -sfT "$PLAYWRIGHT_NPM_PATH" node_modules/playwright

`browser.test.js` can still filter WebKit out when `PLAYWRIGHT_SKIP_WEBKIT=1`
is set, but Linux no longer sets it. The JavaScript library test commands run
Node browser tests through `PLAYWRIGHT_FHS` when available.


FUTURE WORK (in priority order)
-------------------------------

1. PLAYWRIGHT VERSION COUPLING (low effort, ongoing):
   Today the Playwright version is governed by the nixpkgs flake input
   (currently nixos-25.11). Bumping the nixpkgs input bumps Playwright
   in lockstep. New Playwright features are only available after nixpkgs
   picks them up. Track this and bump the channel when needed.

2. COLD-PATH WEBKIT MYSTERY (resolved, but worth a note):
   Earlier (apt-based) cold-path runs hit a different WebKit failure:
   "page.goto: WebKit encountered an internal error". Apt-package-version
   diff showed identical packages between warm and cold images. Never
   diagnosed. Now obsolete because the apt path is gone, but watch for
   similar surprises when bumping nixpkgs Playwright.

3. macOS HOST INVOCATION:
   Today, running `cd javascript && nix develop --command task setup`
   on a developer macOS laptop pulls Linux-shaped Playwright browsers
   from nixpkgs — generally fine for a Linux container, undefined on
   bare macOS. Browser tests are container-only. Document if anyone
   asks; otherwise leave as-is.

4. OFFLINE OPERATION:
   With prefer-offline=true and one successful setup, the lifecycle
   works without network for all three languages. True --network=none
   may still hit edge cases (npm registry verification on `npm ci` if
   cache is stale). Worth a future test pass with a fully populated
   container cache.
